Bank datacenter migration: 100+ AIX & Linux servers, P6 → P770, EMC VMAX

The challenge

What I did

HPC · Engineering Thermal HPC cluster deployment — 12-node ANSYS Fluent CFD +

Business problem

The thermal engineering team needed a managed ANSYS Fluent environment for large CFD simulations. The jobs were far too big for individual workstations and required dedicated HPC resources — a job scheduler, shared storage mounted across the workstations, head node and all compute nodes, and a way to run the ANSYS Fluent GUI on the engineers’ own GPU-equipped Linux workstations via remote desktop.

Architecture

My responsibilities

  • Installed and standardized RHEL across the head node and all compute nodes
  • Configured the PBS Pro scheduler — queues, fair-share, and job distribution across the cluster
  • Integrated FlexLM licensing for ANSYS
  • Installed and configured ANSYS Fluent
  • Built the remote-visualization path so the Fluent GUI renders on the engineers’ GPU-equipped Linux workstations (the cluster nodes are compute-only)
  • Enabled NoMachine / XRDP for remote engineer sessions

Issues solved

  • XRDP black screen — corrected the session/display-manager startup so the remote desktop rendered instead of a blank screen
  • Wayland / Xorg conflicts — forced an Xorg session for compatibility with the GPU and remote-viz tooling
  • GLIBCXX library mismatch — aligned the libstdc++ / toolchain version to what ANSYS required
  • MPI bootstrap failures — fixed inter-node launcher / SSH and fabric config so MPI ranks start cleanly across all nodes
  • FlexLM connectivity — opened the correct license ports and pointed clients at the license server

Outcome

  • 12-node cluster fully operational
  • Engineers running production CFD jobs end to end — submit, compute, and visualize remotely

Security & Compliance Hardening RHEL to the CIS Benchmark (Level 1) +

How I take a Red Hat estate from an unknown security posture to a measured, audit-ready CIS baseline — and keep it there.

Approach

  1. Baseline & scope — inventory hosts, pick the CIS profile (Level 1 for servers), and record justified exceptions with named owners.
  2. Scan first — run OpenSCAP against the CIS profile to get a measured starting score before touching anything.
  3. Filesystem & boot — disable unused kernel modules/filesystems, set nodev/nosuid/noexec mount options, lock down bootloader permissions.
  4. Services & network — remove unneeded packages/services, harden sysctl (redirects, IP forwarding), tighten firewalld zones.
  5. SSH & access — key-only auth, no root login, approved Ciphers/MACs, idle timeout, and PAM password-quality + lockout policy.
  6. Audit & logging — enable auditd with the CIS rule set, make logs immutable, and forward to the SIEM.
  7. Automate & enforce — codify every control in an Ansible role, re-scan to prove remediation, and schedule drift detection.
  8. Evidence — export OpenSCAP reports mapped to controls as audit evidence.
Same pattern maps cleanly to NIST 800-53 / DISA STIG profiles.

Migrations Zero-downtime SAN migration with Storage vMotion +

Migrating a live VMware estate from an old SAN to a new one with no maintenance window — every VM moved online.

Runbook

  1. Discover & plan — map every datastore → VM → LUN, capture capacity and IOPS, and confirm the target (zero downtime).
  2. Stand up the new array — provision LUNs, zone the fabric / configure iSCSI, and present storage to every host.
  3. Network paths — add dual-port 10GbE iSCSI NICs with multipathing; validate there is no single path before proceeding.
  4. New datastores — create and verify VMFS datastores on the new array across the whole cluster.
  5. Live migrate — Storage vMotion VMs in waves, lowest-risk first, watching latency and throughput throughout.
  6. Validate — confirm each VM is healthy on new storage; spot-check applications and databases.
  7. Decommission — unmount old datastores, remove stale paths/zoning, and return the old array.
Run this way, the estate moves with zero user impact — everything stays online.

Cloud & IaC Azure landing zone with Bicep +

A secure, governed Azure landing zone expressed entirely as code — the reference design I deploy and harden.

Approach

  1. Governance — management groups with separate prod / non-prod subscriptions; RBAC and Azure Policy applied at the group level.
  2. Network — hub-and-spoke VNets, NSGs per subnet, private endpoints; nothing exposed publicly by default.
  3. Identity — Entra ID SSO, least-privilege roles, and PIM for just-in-time elevation.
  4. Secrets — Key Vault for every secret; nothing hardcoded in templates or pipelines.
  5. Infrastructure-as-code — the whole zone in Bicep modules, deployed via CI/CD with what-if previews.
  6. Protect — Azure Backup (Recovery Services Vault) with tested file-level and whole-VM restore runbooks.
  7. Cost — scheduled auto-shutdown of non-prod and idle systems.

Platform & HPC GPU-accelerated HPC cluster with PBS Pro +

Standing up a GPU-accelerated HPC cluster engineers can actually use — including remote 3D visualization that standard remote-desktop tools can’t deliver.

Design

  1. Topology — login/head node plus CPU compute nodes, a shared parallel filesystem mounted across workstations/head/nodes, and AD-integrated auth (SSSD / Kerberos).
  2. Scheduler — PBS Pro with fair-share queues and separate GPU vs CPU partitions.
  3. GPU remote viz — engineers run the interactive GUI (CFD / FEA) on their own GPU-equipped Linux workstations, with the heavy compute running headless on the cluster.
  4. Node imaging — golden images built with Packer; configuration and state managed by SaltStack for reproducible nodes.
  5. Job flow — batch submission with a batch-to-GUI bridge, plus automated file transfer in and out.
  6. Identity — AD logins map straight to cluster accounts, so users hit no friction.
